# DDDとは？
DDDは、「技術」ではなく、「業務 (ドメイン)」を中心にソフトウェア設計をする手法のこと。

# DDDの思想
### DDD専用の用語について

#### ① ユビキタス言語
エンジニアとビジネスサイド(企画者など)が「共通の言葉」を使うこと。コードの中の変数名と企画書の中の言葉を完全に一致させるための言語。

ユーザーがマッチングした時、、、、
これをコード上でもUserクラス、Match()メソッドと名付ける。

#### ② 境界づけられたコンテキスト (Bounded Context)
言葉の意味が通じる「範囲」を明確に区切ること。巨大なシステムでは、同じProductという言葉でも、部署によって意味が変わってくる。

- ECサイトの例：
    - 配送担当にとっての「商品」：重さ、サイズ、在庫場所が重要になる。
    - 販売担当にとっての「商品」：価格、説明文、セール期間が重要になる。

人によって商品の解釈が異なる。
これをDDDは、無理に1つのProductクラスにまとめることはせず、「配送コンテキストのProduct」と「販売コンテキストのProduct」を別々のクラスとして作る。


# DDDの実装部分
DDDで実装する部品のこと

① エンティティ (Entity)
IDで区別されるオブジェクトのこと。中身のデータが変わっても、IDが同じなら同じものとして扱う。

例：
Userの名前を変えたとしても、IDが123ならそれは同じユーザーである。

② 値オブジェクト (Value Object)
値そのもので区別されるオブジェクト。不変であり、値を変更するときは新しいオブジェクトを作る。

例：1万円
金は、製造番号IDや1万円というお金としての価値(Value)がある。

#### エンティティ(識別が必要なもの)：通帳、口座

- 銀行口座
    - ID：口座番号
    - Value：残高 (1万円、10万円)

残高が1万円から10万円に変わっても、その口座番号は同じである。識別子があるからこそどの口座かが分かる。

#### 値オブジェクト (識別が不要なもの)：お金の価値

- 1万円という価値
    - Aさんが、財布C,Dを持っている時、Cに入っている1万円をDに移しても、Aの資産価値は変わらない。
    - システム上で1万を支払う場合、その1000がどの製造番号のお札かまでをシステムが知る必要はない。あくまで、お金の価値が1万ですよという価値のみを知っていればいい。つまり、IDが不要で値の価値(1万、10万)で区別が行えるということ。

③ 集約 (Aggregate)
必ず整合性を保ちたいデータの塊のこと。トランザクションで一単位になる。

- 例：Order(注文)とOrderItem(注文明細)
    - 注文明細だけを勝手にDBに保存して、親の注文が存在しないという状況は許されない。
    - 必ずOrderを通してOrderItemを操作するように制限する。

④ リポジトリ (Repository)
集約を保存したり、取り出したりするDBの永続化処理をinterfaceで抽象化する役割。

⑤ ドメインサービス (Domain Service)
モノ (Entity/Value Object)に持たせるには不自然な「操作、アクション」を引き受ける場所のこと。

例：送金処理
銀行口座Aが、口座Bに送信するとなってしまっている場合、
口座自体が他の口座に送信するのは役割を持ちすぎ。口座はあくまで口座番号という識別子といくらお金があるかという価値の情報を持っているだけのオブジェクトとしての役割のみを持つべき。

ここで、「ドメインサービス」を活用して送信サービスという一つの処理として切り離す。
これによって、「送信サービスが、口座AからBへ送金を実行する」という口座(エンティティ)と送金処理(ドメインサービス)という切り分けをうまく行える。

# なぜDaburi ZeroでDDDを使うのか？
商品という言葉1つとっても、文脈によって意味が異なります。
- 買い物中のユーザー: 「これ、家にあるやつと同じ（似ている）か？」という比較対象。
- 在庫管理システム: 「名前、画像、登録日」を持つ記録データ。
- AI解析サーバー: 「512次元の浮動小数点数」という数値ベクトル。

これらを整理せずに実装すると、コードがカオスになります。DDDを用いることで、これらを「ユビキタス言語」として定義し、正しく切り分けます。


# DDDの概念とDaburiZeroでの実例
① ユビキタス言語（共通言語）
開発者が利用者と同じ言葉を使うことで、認識のズレをなくすことを目的にします。
- Embedding (特徴量ベクトル): 画像をAIで数値化したもの。
- Similarity (類似度): 2つの商品がどれくらい似ているかを示す指標。
- Item (在庫アイテム): ユーザーが管理している1つの備品。

② エンティティ (Entity)
ID(識別子)で区別されるオブジェクト。
- User: 名前が変わっても、ユーザーIDが同じなら同一人物になる。
- Item: 商品名(Name)を「生搾り醤油」から「キッコーマン醤油」に書き換えても、登録IDが同じならそれは同じものとして考える。

③ 値オブジェクト (Value Object)
値そのものに意味があり、IDを持たない普遍のオブジェクト。
- Embedding（ベクトル量）: 512個の数字の並び。これ自体に「ID」は不要で、数値が変わればそれは別のベクトル。
- Quantity（数量）: 「3個」という数値。

④ ドメインサービス (Domain Service)
モノ(Entity)に持たせるには、不自然なアクションを引き受ける場所です。
- 在庫Aが自分自身で「新しい写真B」と比較して、似ていると判断するのは不自然。あくまで在庫Aは、こういう情報を取り合っていますよに留めておくべき。
- ここで、ドメインサービスという考え方を活用して、「照合サービス」という一つとして切り出す。照合サービスが、在庫データと撮影データを受け取り、その類似度を計算して持っているか、持っていないかの判定を下すという役割を果たします。在庫Aの要素が絡むけど、在庫Aがその処理を担うのはおかしいよなと思われる時に、このドメインサービスという考え方が生きます。

# 「Daburi Zero」における3ステップの実装アプローチ
STEP1: ドメインオブジェクトの特定
- エンティティ: User、Item(在庫商品)
- 値オブジェクト: ItemEmbedding(512次元ベクトル)、ImageURLなど
- ドメインサービス: SimilarityCalculator(ベクトル間のコサイン類似度計算)

STEP2: 境界づけられたコンテキストの定義
- 在庫管理コンテキスト: 商品の登録、名前の変更、ストックの有無を扱う。
- 画像解析コンテキスト: GoがPythonへ画像を送り、ベクトルに変換してもらう。

STEP3: クリーンアーキテクチャへの肉付け
DDDで定義した「商品の類似度を判定する」という最も重要なビジネスルール（Domain層）を、中央に配置します。その周りをクリーンアーキテクチャの「壁」で囲むことで、PostgreSQLのSQLの書き方や、Pythonサーバーとの通信プロトコル（gRPC）といった外部の技術的な詳細が、ビジネスルールを汚さないように隔離・保護します。
