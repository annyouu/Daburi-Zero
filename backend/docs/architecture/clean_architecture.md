# クリーンアーキテクチャとは？
クリーンアーキテクチャとは、「関心事の分離」を目的としたアーキテクチャパターンです。
ソフトウェアをいくつかの層に分け、依存の方向は常に内側(ビジネスロジック)へ向かうという絶対的なルールを設ける。これにより、データベースやフレームワークといった外部の技術的な詳細が変更されても、システムの核となるビジネスロジックが影響を受けないように設計します。

# 今回の例でクリーンアーキテクチャの構造をどう組み込むか
本プロジェクト「Daburi Zero」では、以下のように層を定義する。
1. Domain (ドメイン層):
- 役割: アプリケーションの最も純粋なビジネスルール (DDDのエンティティや値オブジェクト)を定義する。
- 依存: どこにも依存しない。
- 例: Itemエンティティ

2. UseCase (ユースケース層):
- 役割: ドメインオブジェクトを操作して、特定の機能 (商品を登録する、在庫を照合するなど)を実現します。
- 依存: Domain層のみに依存する。

3. Presentation / Interface Adapter（プレゼンテーション層）
- 役割: 外部（フロントエンドやHTTPリクエスト）とユースケースを繋ぎます。
- 依存: Usecase層に依存する。
- 例: item_controllerなど

4. Infrastructure（インフラストラクチャ層）
- 役割: DB操作、gRPC通信、ストレージ保存などの具体的な技術実装を担う。
- 依存: Usecase層が定義したインターフェース(Repository等)に依存する。
- 今回の例: pgvectorを使ったSQL実行、PythonへのgRPCクライアント、MinIOへのアップロードなど。

# 実装上の構成要素
- 依存性逆転: 
通常、UsecaseはDBを操作したいと考えますが、直接Infrastructureに依存すると「壁」が壊れます。そのため、Usecase層でインターフェース（Repository）を定義し、Infrastructure層がそれを実装することで、依存の向きを内側へ逆転させます。

# クリーンアーキテクチャとDDDの違い
この2つは対立するものではなく、「補完関係」にあると思っています。
| 概念 | 焦点 | 役割 |
|-----|-----|-----|
|DDD|中身の正しさ|複雑な業務ルール（ドメイン）をどう整理し、どうプログラムで表現するかを決める。|
|クリーンアーキテクチャ|外枠の整合性管理|フォルダ構成や配置を考え、層同士をどう繋げるか、依存の向ける先をどうするか、詳細な技術からドメインを切り離して保護する。|

## 本プロジェクトでの使い分け
- DDDを使って、「在庫と撮影画像をどう比較して『ダブリ』と判定するか」という中身を定義していきます。
- クリーンアーキテクチャ を使って、その中身を「PythonのgRPCサーバー」や「PostgreSQL」といった具体的な技術から切り離して保護し、変更したとしても影響がないようにします。
